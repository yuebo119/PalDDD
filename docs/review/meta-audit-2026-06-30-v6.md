### 4.1 P1/P2/P3 与实际危害的映射

| 任务 | 优先级 | 实际危害 | 修复类型 | 是否合理 |

| ITM-039（SoftDelete 安全文档） | P2 | 低（已有文档的方法补注） | 文档 | ⚠ P2 偏高 |

### 4.2 分类标准分析

当前分类标准（P1=近期修复/P2=中期增强/P3=长期演化）基于**时间紧迫度**而非**危害程度**。这导致：

- **ITM-038（连接泄漏）与 ITM-037（4 行 editorconfig）同为 P2**——危害差一个数量级，但优先级相同
- **ITM-032（README 数字）被标为 P1**——原因是"容易修"，而非"危害大"
- **ITM-031（CA1031）被标为 P1**——原因是"涉及 27 处修改"的规模感知，而非危害评估

### 4.3 改进建议

```
P 分级改为危害 × 复杂度双维度:

       高危害   中危害   低危害
易修复   P0       P1      P2
中等     P1       P2      P3
难修复   P2       P3      评估

判断危害的依据:
- 高危害: 数据损失/安全漏洞/资源泄漏/运行时崩溃/编译失败
- 中危害: 不一致/潜在的并发风险/架构退化
- 低危害: 注释/文档/装饰性/IDE 噪音
```

按此标准，ITM-038 应为 P1、ITM-037/039 应为 P3、ITM-032 评估后废弃。

## 段 5：方法论评估

### 5.1 工具选择的适当性

| 工具 | 使用场景 | 适当？ | 误判关联 |
|------|----------|:------:|----------|
| `grep` + 计数差异 | M4（异常过滤"不一致"判断） | ❌ 工具误用 | grep 输出的是文本匹配次数，不是语义判断 |
| `grep` + 行号定位 | 定位 catch(Exception) 位置 | ✅ 适当 | 无 |
| Serena `find_symbol` | 提取完整方法体 | ✅ 适当 | 但 M1/M2 未使用此功能 |
| Serena `get_symbols_overview` | 文件级符号概览 | ✅ 适当 | 无 |

### 5.2 评审方法论对比：补充卷 II vs v4

| 维度 | 补充卷 II | v4 评审 |
|------|----------|---------|
| 审计方式 | **逐行审计** | 抽样 + 概括 |
| 聚焦点 | SQL 注入、线程安全、技术债 | 架构设计、DDD 原则、性能契约 |
| 结论形式 | ✅ 零风险 / ⚠ 调用方控制 / ❌ 需修复 | P1/P2/P3 建议 + 评分 |
| 不确定项标注 | 明确标注"⚠" | 无标注 |
| 发现类型 | 实现层缺陷（SQL 注入） | 架构层建议（注释统一、AOT 配置） |
| 准确率 | **100%** | 43% |

**结论**：补充卷 II 的方法论（逐行审计 + 风险标注 + 聚焦实现）显著优于 v4 评审（抽样概括 + 评分 + 聚焦架构）。这不是"哪个评审者更好"的问题，而是**两种不同的审计策略**——实现驱动 vs 架构驱动。前者准确率更高，因为实现层的事实是可逐行验证的"是/否"，而架构层的判断依赖主观评分和概括。

### 5.3 工具组合的最佳实践

基于准确率数据，最优工具组合为：

```
1. review-snapshot.sh → 锚定基线（防 R3）
2. Serena find_symbol（完整方法体）→ 逐行审计（防 R1）
3. grep（定位，不做计数差异判断）→ 辅助定位（防 R4）
4. dotnet build → 验证分析器行为（防 R5）
5. ls + git show → 验证路径/引用（防 R6）
```

### 7.2 ITM-031 的隐藏副作用（详细分析）

```
修改前:
  Directory.Build.props 全局 NoWarn 含 CA1031
  理由：后台处理器必须隔离任意异常，每处带 Justification
  → 所有 catch(Exception) 被全局放行
  → 风险：非后台代码的 catch-all 也被放行

修改后:
  移除全局 NoWarn 中的 CA1031
  5 处裸 catch(Exception) 添加 [SuppressMessage]
  → 22 处带 when(ex is not OperationCanceledException) 的 catch 未加
  → 当前: .NET 11 Preview 5 的 CA1031 不触发带 when 的 catch ✅
  → 风险: Microsoft 官方文档 CA1031 未提及 when 豁免行为
       若未来版本收紧 → 22 处 → 编译失败 (TreatWarningsAsErrors=true)
```

**风险量化**：
- 概率：低（.NET 大概率不会改变现有行为）
- 影响：若发生，22 处编译错误需要逐一加 [SuppressMessage]
- 修复成本：约 30 分钟（加 22 个属性）
- **判定**：低概率 × 中等修复成本 = 可接受的残留风险。但关键是——**这个风险在修改时未被识别**，这是一个流程缺陷而非技术缺陷。

### 7.3 ITM-038 的测试缺失

```
PostgreSqlReadWriteRouter 修改:
  + 实现 IAsyncDisposable
  + 添加 DisposeAsync() 方法
  + DI 注册为 Singleton（容器自动调用 DisposeAsync）

缺失的测试:
  - Dispose 后调用 GetReader() 应抛 ObjectDisposedException
  - Reader 为 null 时 DisposeAsync 不崩溃
  - 集成测试验证 DI 容器关停时实际调用了 DisposeAsync
```

这是 17 项任务中**价值最高但交付质量最不完整的修复**。修复本身正确（基于 DI 容器的 Singleton dispose 机制），但缺乏回归测试使其在未来的重构中容易退化。
