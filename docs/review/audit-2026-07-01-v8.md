### 2.2 发现清单

**本次审计未发现 P0/P1/P2 级别问题。** 以下为 6 项评估性观察（OBS），均为设计层面的取舍记录，非缺陷。

#### ⚪ OBS-01 — Saga 补偿顺序：失败步骤优先于已执行步骤逆序

**证据**：`src/PalDDD.Transactions/SagaCompensation.cs:56-66`（子代理读取）

```csharp
List<SagaStep> targets = [];
foreach (var key in state.ExecutedStepKeys)  // 按执行顺序
    ...
if (failedStep.CompensateAsync is not null && !targets.Contains(failedStep))
    targets.Add(failedStep);  // 失败步骤追加到末尾
// Backward 策略从 targets.Count-1 逆序执行 → 失败步骤第一个被补偿
```

**论证**：当前实现实际是"失败步骤 → 已执行步骤逆序"，与注释"补偿按实际已执行步骤的逆序执行"存在微妙偏差。但语义上合理——失败的步骤可能部分执行，应优先回滚。

**建议**：更新注释，明确补偿顺序为"失败步骤优先 → 已执行步骤逆序"。纯文档级，无行为变更。

#### ⚪ OBS-02 — Saga 非线程安全（设计约束，非缺陷）

**证据**：`src/PalDDD.Transactions/Saga.cs:108-109`（子代理读取）

`GetFrozen()` 中 `_frozen ??= _stepsByKey.ToFrozenDictionary(...)` 非原子操作。若两线程同时首次调用 `ProcessEventAsync`，可能同时构建 FrozenDictionary。

**论证**：文档已明确"在构造函数中注册，运行时只读"。这是设计约束而非缺陷——Saga 实例通常是 Scoped 生命周期，单请求内不会并发调用。但缺少编译期或运行时守护防止误用。

**建议**：考虑在 `ProcessEventAsync` 首次调用后 freeze 整个 Saga 实例（不仅冻结字典），后续注册抛 `InvalidOperationException`。评估类，非必须。

#### ⚪ OBS-03 — Saga 补偿不强制经过 Outbox

**证据**：`src/PalDDD.Transactions/SagaCompensation.cs`（子代理读取）

补偿动作 `step.CompensateAsync` 是直接委托调用，未经过 Outbox。若补偿需发布事件（如"库存已恢复"），补偿委托的实现者需自行写入 Outbox。

**论证**：这是设计选择——框架不强制补偿路径的事件发布策略，给予实现者灵活性。但若实现者遗漏 Outbox，补偿事件可能丢失。

**建议**：在 `SagaStep.CompensateAsync` 的 XML doc 中增加 `<remarks>` 提示"补偿动作若需发布事件，应通过 Outbox 保证至少一次语义"。纯文档级。

#### ⚪ OBS-04 — Entity&lt;TId&gt;.Id 的 init setter 与 EF Core 反序列化

**证据**：`src/PalDDD.Core/Entity.cs:68` `public TId Id { get; init; }`（子代理读取）

**论证**：EF Core 8+ 支持 `init` 属性的设置（通过反射绕过 init 限制）。但其他 ORM 或序列化框架可能无法设置 init 属性。鉴于框架双 ORM 策略中 Dapper 不依赖属性设置器（直接 SQL 映射），且 EF Core 8+ 兼容，实际风险低。

**建议**：在 `Entity<TId>` XML doc 中注明"Id 的 init setter 要求 ORM 支持 init 属性设置（EF Core 8+ 兼容）"。评估类。

#### ⚪ OBS-05 — ValueObject&lt;T&gt; 无基类不变量校验

**证据**：`src/PalDDD.Core/ValueObject.cs:59`（子代理读取）

`ValueObject(T value)` 构造函数不做任何验证，完全依赖派生类型。

**论证**：文档已说明"业务值域约束应由派生类型自行实现"。这是合理的——值对象的校验逻辑因类型而异（如温度不能低于绝对零度、百分比 0-100），基类无法预知。但缺少 `IValueObject` 的 `Validate()` 契约方法供派生类型约定。

**建议**：评估是否在 `IValueObject` 增加 `bool IsValid()` 或在 `ValueObject<T>` 构造函数接受可选验证委托。评估类，当前"派生类型自校验"方案可接受。

#### ⚪ OBS-06 — .NET 11 Preview 依赖的兼容性窗口

**证据**：`global.json` 锁定 `11.0.100-preview.5`，`<Features>runtime-async=on</Features>`（read_file 实测）

**论证**：框架依赖 .NET 11 Preview 的 `static abstract`、`JsonSerializerContext` 源生成增强、`runtime-async` 等特性。Preview API 可能在 RTM 前发生破坏性变更。ADR-013 已记录迁移计划，ITM-041 待 GA 后执行。风险已知且有缓解措施。

**建议**：维持 ADR-013 计划。在 .NET 11 RC 阶段提前验证破坏性变更。评估类。

## 第四部分：架构亮点深度论证

> 本部分对项目中最值得称道的 5 个架构决策进行充分论证，说明"为什么好"而非仅"好"。

### 4.1 DIM 桥接消除 MakeGenericType（核心技术亮点）

**设计**：`ICommandHandler<TCommand,TResponse>` 继承非泛型 `IHandler`，通过 Default Interface Method 提供编译时桥接：

```csharp
// CommandHandler.cs:50-58 — DIM 桥接，零运行时反射
ValueTask<object?> IHandler.HandleAsync(IBaseRequest request, CancellationToken ct)
{
    var task = HandleAsync((TCommand)request, ct);
    if (task.IsCompletedSuccessfully)
        return new ValueTask<object?>(task.Result);    // 同步完成零分配
    return AwaitAndBox(task);                            // 异步路径按需分配
}
```

**为什么好**：
1. **AOT 安全**：Dispatcher 只持有 `IHandler` 引用调用 `HandleAsync(IBaseRequest, CT)`，完全不需要 `MakeGenericType` 将 `IHandler` 转为 `IHandler<T>`。这是 100% Native AOT 兼容的关键。
2. **零分配快速路径**：`IsCompletedSuccessfully` 检查后直接 `.Result`，同步完成时不构造异步状态机。仅异步路径通过 `AwaitAndBox` 按需分配。
3. **类型安全**：`(TCommand)request` 强制转换由 Dispatcher 的 `Type` 键查找保证类型一致，设计上不可能路由错误。
4. **可组合**：PipelineBehavior 采用同模式 DIM 桥接，管道行为链与 Handler 链正交组合。

**对比替代方案**：传统 MediatR 使用 `MakeGenericType` + `MethodInfo.Invoke`，在 AOT 下需要保留反射元数据，破坏 trimming。Pal.DDD 的 DIM 方案从根本上消除了这一需求。

### 4.2 单链表事件存储 + ref struct 枚举器（性能契约）

**设计**：`Entity` 用 `_head`/`_tail` 单链表存储领域事件，`DomainEventEnumerable` 是 `ref struct` 枚举器：

```csharp
// Entity.cs — 单链表追加，O(1)，零容器分配
public void RaiseEvent(DomainEvent domainEvent)
{
    if (_head is null) _head = domainEvent;
    else _tail!.Next = domainEvent;
    _tail = domainEvent;
}
```

**为什么好**：
1. **零堆分配**：无事件时零分配（空链表），有事件时仅事件本身分配（无 List 容器开销）。
2. **ref struct 枚举器**：`DomainEventEnumerable` 和 `DomainEventEnumerator` 都是 `ref struct`，栈分配，零装箱。
3. **不可变契约**：`conventions.md §12.1` 明确"不可改为 `IEnumerable<T>`"——这是性能契约，架构测试虽未直接断言但文档强制。

**对比替代方案**：`List<DomainEvent>` 在空实体时仍分配 72 字节（空 List 内部数组），高频实体创建场景 GC 压力显著。单链表方案在无事件时零分配。

### 4.3 FrozenDictionary 冻结模式（并发安全契约）

**设计**：`Dispatcher` 运行时增量注册 Handler 到 `Dictionary`，首次分发时 `ToFrozenDictionary()` 冻结为只读：

```csharp
// Dispatcher.cs — Freeze() 后 FrozenDictionary 不可变
public void Freeze() => _frozen = _stepsByKey.ToFrozenDictionary(...);
```

**为什么好**：
1. **O(1) 查找 + 零 GC**：`FrozenDictionary` 是 .NET 8+ 的只读字典，查找性能优于 `Dictionary`，无运行时哈希桶调整。
2. **线程安全**：冻结后不可变，多线程读取无需锁。
3. **生命周期清晰**：`HandlerRegistrar`（IHostedService）在启动时注册，然后 `Freeze()`，运行时只读。

**对比替代方案**：`ConcurrentDictionary` 提供线程安全但查找有锁开销且允许运行时修改（破坏不可变性契约）。`FrozenDictionary` 冻结模式是"启动时写、运行时读"场景的最优解。

### 4.4 架构边界测试动态扫描（质量工程亮点）

**设计**：`InfrastructureAdapters_AreExplicitlyNonAot` 动态扫描所有 csproj，而非硬编码 Theory 列表：

```csharp
// ArchitectureBoundaryTests.cs:217-250 — 动态扫描，防止漏检
var csprojFiles = Directory.EnumerateFiles(Path.Combine(Root, "src"), "*.csproj", ...);
foreach (var csprojPath in csprojFiles)
{
    if (!project.Contains("<IsAotCompatible>false</IsAotCompatible>")) continue;
    // 断言三属性齐全
    Assert.Contains("<IsTrimmable>false</IsTrimmable>", project);
    Assert.Contains("<VerifyReferenceAotCompatibility>false</...>", project);
}
Assert.True(checkedProjects >= 8, "动态扫描应至少检查 8 个业务 false 项目");
```

**为什么好**：
1. **防漂移**：新增非 AOT 项目自动被检测，无需手动更新 Theory InlineData。
2. **下限断言**：`Assert.True(checkedProjects >= 8)` 防止扫描逻辑空转（如目录结构变更导致扫描不到任何项目）。
3. **历史教训内化**：ITM-033 修复前是硬编码 5 个项目的 Theory，漏检 7 个。动态扫描是误判模式 7（grep 表面 ✅ → 跳过深度读取）的系统性防范。

### 4.5 编译期 DDD 治理（战略 DDD 落地）

**设计**：`StrategicDddAnalyzer` 15 条诊断规则（PDDD001-015）在编译期防止架构腐化：

- 领域模型必须声明 `[BoundedContext]`
- 领域事件必须 `sealed` + `[GenerateMessage]`
- 消息名必须含 BC 前缀 + `.v{N}` 后缀
- ProcessManager / ProjectionHandler 必须 `sealed` + 归属 BC

**为什么好**：
1. **编译期强制**：违反 DDD 约束在编译时报错（TreatWarningsAsErrors），而非运行时或代码审查时发现。
2. **三位一体**：分析器规则（PDDD001-015）↔ AI 提示模板（`.pal/prompts/`）↔ 架构测试，形成编译期-AI-测试三层防护。
3. **战略 DDD 落地**：多数 DDD 框架仅提供战术构建块（Entity/ValueObject），Pal.DDD 通过分析器强制战略约束（BC 划分、消息版本化），这是显著差异化。

## 审计结论

Pal.DDD 是一个**架构成熟度达到生产级**的 DDD/CQRS 框架。其核心价值在于：

1. **零反射 + AOT-first** 的技术选型在 .NET DDD 框架生态中具有显著差异化
2. **编译期治理（15 条分析器）+ 运行时架构测试（25+ 断言）+ AI 提示模板** 三位一体的质量保障体系
3. **DIM 桥接 / FrozenDictionary 冻结 / 单链表事件存储 / PipelineStateMachine** 等性能契约设计体现了对 .NET 运行时特性的深刻理解
4. **13 章 conventions.md + 14 个 ADR** 的文档完备度使架构决策可追溯、可审计

**综合评分 8.6/10 · 无保留意见**。6 项观察均为设计取舍记录，不影响生产可用性。项目经 9 轮审计迭代后处于维护稳态，建议后续聚焦 .NET 11 RTM 迁移（ADR-013）和 Benchmark 实测（ITM-041）。
