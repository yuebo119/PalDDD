## 二、错误分类详解

### 类别 A：误判（假阳性）— 评审声称有问题，实际无问题

#### A1. `InMemoryOutboxStore.RequeueDeadAsync` 并发语义误判 [高]

- **来源**：`serena-comprehensive-review-2026-06-29-v4.md` P2 建议
- **错误内容**：声称"状态转换未在锁内完成，存在并发竞态"，建议"确保完整状态转换在锁内完成"
- **事实**：完整状态转换（Status/ProcessedAt/Error/NextAttemptAt/LockedBy/LockedUntil）**已在 `lock (_lock)` 内完成**
- **发现方式**：`action-items-2026-06-29-v2.md` 事实核查修正段
- **根因**：评审时只读了代码片段（`lock` 块的开头），未读完整 `lock` 块体，基于不完整信息下结论

#### A2. `PeriodicBackgroundProcessor` 异常隔离误判 [中]

- **来源**：`serena-comprehensive-review-2026-06-29-v4.md` P2 建议
- **错误内容**：声称"未过滤 `OperationCanceledException`，优雅关停时会误报错误日志"
- **事实**：已正确过滤关停场景（`when (stoppingToken.IsCancellationRequested)`）。仅"非关停取消"未被过滤，属设计决策而非缺陷
- **发现方式**：`action-items-2026-06-29-v2.md` 事实核查修正段，降级为 P3
- **根因**：评审时未区分"关停取消"与"非关停取消"两种语义场景，将设计决策误判为缺陷

#### A3. `DapperProjectionCheckpointStore` 缺失误判 [中]

- **来源**：`serena-comprehensive-review-v2.md` 前期评审记录
- **错误内容**：声称"Dapper 路径缺少 `IProjectionCheckpointStore` 实现"
- **事实**：`DapperProjectionCheckpointStore` 已存在
- **发现方式**：`serena-comprehensive-review-v2.md:257` 自修正标注"前期评审记录已过期，当前版本已补齐"
- **根因**：评审基于过期版本快照，未在当前 commit 上重新验证

#### A4. 异常过滤"不一致"初判 [低]

- **来源**：本轮评审（2026-06-30）维度 4 健壮性分析
- **错误内容**：grep 显示 27 处 `catch(Exception)` vs 11 处 `when(ex is not OperationCanceledException)`，初判为"异常过滤不一致，存在规范执行缺口"
- **事实**：全部合规——其余 16 处采用"前置 `catch(OperationCanceledException){throw;}` + `catch(Exception)`"模式
- **发现方式**：实施时逐文件核查 catch 上下文后修正
- **根因**：grep 计数差异未经语义核查就下结论，未识别两种等价的合规模式

### 类别 B：过度预估 — 工作量/影响范围严重高估

#### B1. ITM-031 "27 处 catch(Exception) 需加 [SuppressMessage]" [高]

- **来源**：`action-items-2026-06-30.md` ITM-031
- **错误内容**：声称需在 27 处 `catch(Exception)` 上方添加 `[SuppressMessage]`，列出 16 个涉及文件，风险评估为"中（涉及 27 处源码修改）"
- **事实**：移除全局 `CA1031` 后，只有 **5 处**触发编译错误（无 `when` 过滤的裸 `catch(Exception)`）。带 `when(ex is not OperationCanceledException)` 的 catch 不触发 CA1031
- **发现方式**：实施时 `dotnet build` 只报 5 处错误
- **根因**：**未理解 CA1031 分析器的触发条件**——该规则只对无过滤的 `catch(Exception)` 报警，带 `when` 子句的 catch 被视为"已处理"不报警。评审时假设"所有 `catch(Exception)` 都触发 CA1031"，这是对分析器行为的错误心智模型

### 类别 C：事实性错误 — 引用的代码事实与实际不符

#### C1. ITM-039 方法名错误 [中]

- **来源**：`action-items-2026-06-30.md` ITM-039（源自用户提供的任务描述）
- **错误内容**：称方法名为 `SoftDelete`/`Restore`/`PurgeOld`
- **事实**：实际方法名是 `Delete`/`Restore`/`Purge`（无 `SoftDelete` 方法，无 `PurgeOld` 方法）
- **发现方式**：实施时读取 `PostgreSqlSoftDelete.cs` 发现方法名不符
- **根因**：任务描述基于记忆/推测而非源码核实，我合并任务时未交叉验证方法名

#### C2. ITM-039 遗漏已有安全文档 [中]

- **来源**：`action-items-2026-06-30.md` ITM-039
- **错误内容**：声称 `SoftDelete`/`Restore`/`PurgeOld` 方法的 XML doc"未标注安全责任"
- **事实**：`Delete` 方法**已有完整安全文档**（`<param name="whereClause">` 含"调用方必须只传入受信任模板，用户输入必须通过 Dapper/Npgsql 参数绑定"）。仅 `Restore` 方法缺失
- **发现方式**：实施时读取源码发现 `Delete` 已有文档
- **根因**：评审未读取目标方法的完整 XML doc，基于"应该没有"的假设下结论

## 四、预防方案

### 方案 1：评审纪律层 — 强制验证协议

| 规则 | 针对根因 | 执行方式 |
|------|----------|----------|
| **完整读取原则**：引用任何代码块时，必须读取完整的方法/块体，禁止基于片段下结论 | R1 | 评审模板增加"已读完整方法体"勾选项 |
| **语义场景区分原则**：涉及异常/取消/并发等场景时，必须列举所有语义子场景再判断 | R2 | 评审模板增加"语义场景清单"段 |
| **当前 commit 锚定原则**：评审开头记录当前 commit hash，所有断言基于该 commit | R3 | 评审报告首行强制记录 commit |
| **grep 语义核查原则**：grep 计数差异必须逐项核查上下文，禁止仅凭数字下结论 | R4 | 评审模板增加"grep 差异已逐项核查"勾选项 |
| **分析器行为核实原则**：声称某分析器规则触发时，必须先 `dotnet build` 验证触发条件 | R5 | 任务清单的"验证"段必须含实际 build 命令 |
| **外部输入交叉验证原则**：用户提供的任务描述中的方法名/类名/行号必须 grep 核实 | R6 | 合并任务时强制 grep 验证关键标识符 |
| **架构测试覆盖度审计原则**：评审架构测试时，必须核查断言数据集是否覆盖全部应覆盖项 | R7 | 评审模板增加"断言覆盖度核查"段 |
| **实测优先原则**：任何数字声明（测试数/文件数/行数）必须实际运行命令获取，禁止采信记忆 | R8 | 任务清单的数字类验证必须附命令输出 |

### 方案 2：工具流程层 — 自动化校验

1. **评审前自动快照**：评审开始时自动运行 `git rev-parse HEAD` + `find src -name "*.cs" | wc -l` 等基线命令，将结果写入评审报告首行，确保所有断言锚定同一快照
2. **任务清单自动验证脚本**：新增 `scripts/verify-action-items.sh`，在任务清单生成后自动校验：
   - 任务描述中引用的方法名/类名是否在源码中存在（`grep -rn "方法名" src/`）
   - 任务描述中引用的行号是否匹配（`sed -n '行号p' 文件`）
   - 涉及分析器的任务是否附 build 验证命令
3. **架构测试覆盖度报告**：新增测试 `ArchitectureBoundaryCoverageTests`，自动扫描 `src/` 下所有 `IsAotCompatible=false` 项目，断言全部出现在 `InfrastructureAdapters_AreExplicitlyNonAot` 的 Theory 数据中（动态扫描，非硬编码列表）

### 方案 3：评审模板层 — 结构化约束

在评审报告模板中增加以下强制段落：

```markdown
## 评审基线
- Commit: <hash>
- 源项目数: <实测值>
- 测试数: <实测值，附 dotnet test 输出>

## 断言核查
- [ ] 所有引用的代码块已读完整方法体
- [ ] 所有 grep 计数差异已逐项核查上下文
- [ ] 所有架构测试断言已核查覆盖度
- [ ] 所有涉及分析器的建议已附 build 验证命令
- [ ] 所有外部输入的方法名/类名已 grep 核实
```
