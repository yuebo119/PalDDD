## 二、核心领域层（PalDDD.Core）

### 2.1 Entity / AggregateRoot

```
Entity (非泛型基类)
  ├── _head / _tail: DomainEvent?    ← 单链表存储领域事件
  ├── RaiseEvent(event)              ← O(1) 追加，protected 封装
  ├── DomainEvents()                 ← 返回 ref struct 枚举器（零分配 foreach）
  └── ClearDomainEvents()            ← O(1) 清空

Entity<TId> : Entity
  ├── Id: TId (where TId : notnull, IEquatable<TId>)
  ├── Equals: 精确类型匹配 + 瞬态不等 + 持久化按 Id
  └── operator == / !=

AggregateRoot<TId> : Entity<TId>
  └── 零附加成员（标记基类）
```

**Serena 符号分析**：
- `Entity` 的 `RaiseEvent` 是 `protected`——只有实体自身可产生事件，DDD 封装正确
- `DomainEvents()` 返回 `DomainEventEnumerable`（`ref struct`），保证零 GC 分配枚举
- `Equals` 使用 `GetType() != other.GetType()` 精确类型匹配——子类不与父类相等，符合 DDD 语义

### 2.2 DomainEvent

```csharp
public abstract class DomainEvent
{
    internal static TimeProvider TimeProvider { get; set; }  // AsyncLocal 测试隔离
    internal DomainEvent? Next { get; set; }                 // 链表节点
    public Guid EventId { get; init; } = Guid.NewGuid();    // 全局唯一
    public DateTimeOffset OccurredOn { get; init; }          // 构造时捕获
}
```

**设计亮点**：
- `AsyncLocal<TimeProvider>` 实现测试隔离——不污染领域方法签名，测试通过 `InternalsVisibleTo` 注入 `FakeTimeProvider`
- `Next` 属性 `internal`——链表结构对领域代码透明
- `init` 语义保证事件创建后不可变

## 三、CQRS 分发层

### 3.1 DIM 桥接架构

```
用户代码层（泛型接口）           框架内部层（非泛型接口）
─────────────────────         ──────────────────────
ICommandHandler<TCmd,TResp> ──→ IHandler       (DIM 自动桥接)
IQueryHandler<TQ,TResult>  ──→ IHandler       (DIM 自动桥接)
IPipelineBehavior<TReq,TRes> → IPipelineBehavior (DIM 自动桥接)
```

**核心机制**：
- `typeof(T)` 编译时常量 → `FrozenDictionary<Type, HandlerEntry>` O(1) 查找
- `IsCompletedSuccessfully` 快速路径 → 同步完成零分配
- `PipelineStateMachine` (~40B) → 替代闭包链 (N×72B)

**Serena `search_for_pattern` 验证**：
- `MakeGenericType` → 生产代码 **0 命中**（仅 8 处注释声明"零 MakeGenericType"）
- `Activator.CreateInstance` → 生产代码 **0 命中**
- `Type.GetType(string)` → 生产代码 **0 命中**
- `Assembly.GetTypes()` → 生产代码 **0 命中**

**结论：CQRS 路径 100% 零运行时反射，完全 AOT 兼容。**

### 3.2 Handler 注册流程

```
AddPalCommandHandler<TCmd,TResp,THandler>()
  → HandlerMarker(typeof(TCmd), typeof(THandler), typeof(TResp), executor)
  → DI: AddSingleton<HandlerMarker>

HandlerRegistrar(IHostedService).StartAsync()
  → Dispatcher.Register(marker) → Dispatcher.Freeze()
  → FrozenDictionary 冻结

运行时: Dispatcher.SendAsync<TCmd>()
  → _handlers.TryGetValue(typeof(TCmd))  // O(1)
  → entry.Executor(scope, entry, ct)      // 直接调用静态方法
```

### 3.3 Pipeline 行为

| 行为 | 功能 | 注册方式 |
|------|------|---------|
| `ValidationBehavior<TReq,TResp>` | 自动调用所有 `IPalValidator<TReq>` | `AddPalPipelineBehaviors()` |
| `LoggingBehavior<TReq,TResp>` | 编译时 LoggerMessage 日志 | `AddPalPipelineBehaviors()` |

## 四、Transactions 层（Outbox / Inbox / Saga）

### 4.1 核心数据模型

**OutboxMessage**（21 个属性）：
- `init` 属性（不可变）：Id, Type, Payload, ContentType, SchemaVersion, CorrelationId, CausationId, TraceParent, TraceState, CreatedAt
- `set` 属性（运行时可变）：Status, ProcessedAt, NextAttemptAt, LockedBy, LockedUntil, RetryCount, Error
- W3C Trace Context 支持分布式追踪透传

**InboxMessage**（9 个属性）：
- `(MessageId, ConsumerName)` 组合唯一约束——同一消息可被不同消费者各处理一次

**SagaState**（12 个属性）：
- `StepStartedAt: Dictionary<string, DateTimeOffset>` ——步骤级精确超时（非从 Saga 创建时间起算）
- `ExecutedStepKeys: Collection<string>` ——精确补偿已执行步骤
- `Version: int` ——乐观并发控制

### 4.3 Saga 编排引擎（`Saga<TState>`，318 行）

```
Saga<TState> (abstract)
├── 数据存储
│   ├── _stepsByKey: Dictionary<string, SagaStep>
│   ├── _stepsInOrder: List<(Key, Step)>  ← 补偿顺序
│   └── _frozen: FrozenDictionary?         ← 惰性冻结
│
├── 策略组件（惰性初始化，组合而非继承）
│   ├── SagaCompensation<TState>           ← Backward / Forward / None
│   └── SagaTimeoutDetector<TState>        ← 步骤级超时
│
├── 配置
│   ├── Clock: virtual TimeProvider        ← 测试可注入
│   ├── MaxRetries (默认 3)
│   └── RetryBackoffPolicy (默认固定 1s，兼容 RetryDelay)
│
└── ProcessEventAsync()
    ├── FindStep: FrozenDictionary O(1) 查找（精确→通配）
    ├── StepStartedAt[stepKey] = Clock.GetUtcNow()
    ├── for attempt in 0..MaxRetries:
    │   ├── step.ExecuteAsync → 成功 → ExecutedStepKeys.Add → return
    │   ├── 失败 + 有重试 → Task.Delay(RetryBackoffPolicy.ComputeDelay(attempt), Clock, ct)
    │   └── 失败 + 无重试 → CompensateExecutedStepsAsync → throw AggregateException
    └── CompensateExecutedStepsAsync: 按策略补偿已执行步骤
```

### 4.4 后台处理器

| 处理器 | 基类 | 周期 | 关键行为 |
|--------|------|------|---------|
| `OutboxBatchProcessor` | `PeriodicBackgroundProcessor` | 可配置 | 原子租约 → 发布 → 标记/重试 |
| `InboxProcessor` | `PeriodicBackgroundProcessor` | 可配置 | CAS 幂等 → 处理 → 标记 |
| `SagaProcessor<TState>` | `PeriodicBackgroundProcessor` | 可配置 | 超时检测 → 补偿 |

### 5.2 RecordedEvent 双构造路径

```
写入路径（EventData 构造）：
  _payload = data.Payload.ToArray()     // 防御性拷贝
  _metadata = data.Metadata.ToArray()   // 防御性拷贝

读取路径（零拷贝构造，internal）：
  _payload = payload                    // 引用赋值，零拷贝
  _metadata = metadata                  // 引用赋值，零拷贝

公共 API:
  RecordedEvent.Rehydrate(ReadOnlyMemory<byte>) → ToArray() → 保守路径
  internal RehydrateFromBytes(byte[]) → 引用赋值 → 零拷贝路径
```

### 5.3 乐观并发

## 十二、端到端数据流追踪

### 12.1 命令处理路径

```
HTTP POST → MapCommand<T>() → Dispatcher.SendAsync()
  → FrozenDictionary.TryGetValue → ExecutePipelineAsync()
    → PipelineStateMachine → ValidationBehavior → LoggingBehavior → THandler.HandleAsync()
      → entity.RaiseEvent(event)
        → SaveChangesAsync() → OutboxDomainEventInterceptor
          → DomainEventCollector.Collect → Serialize → OutboxStore.AddMessage
            → [同一事务提交]

OutboxBatchProcessor（后台）
  → LeasePendingMessagesAsync → Deserialize → Broker.PublishAsync
    → KafkaBroker / RabbitMqBroker.PublishCoreAsync
      → MarkProcessed / ReleaseForRetry
```

### 12.2 消息消费路径

```
Broker 接收 → Deserialize → InboxProcessor.ProcessAsync
  → InboxStore.TryStartProcessingAsync (CAS 幂等)
    → null → Skipped
    → 非null → handler(message) → MarkProcessed / MarkFailed
```

### 12.3 事件溯源路径

```
ProjectionRebuilder.RebuildAsync
  → CheckpointStore.ResetAsync
  → EventLog.ReadAllAsync
    → foreach RecordedEvent:
      → ProjectionProcessor.ProcessAsync
        → CheckpointStore.GetOrAddAsync (幂等)
        → handler.ProjectAsync(message)
```
