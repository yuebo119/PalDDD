# 评审质量定性审计报告（2026-06-30）

> 审计范围：7 份评审报告 + 3 份任务清单 + 14 个 ADR + 全部代码修改
> 审计标准：**仅定性错误**——排除全部计数漂移，只审误判/遗漏/矛盾/错误建议/副作用
> 审计方法：逐项回溯 ITM 任务实现代码，分析每个修复的真实价值与潜在副作用
> 前序声明：本报告不涉及"83→71"等装饰性数字——那类问题已完结

---

## 一、核心发现：评审→修复的价值转化率仅 29%

纵观 3 份任务清单（ITM-025~041，共 17 项任务），按其实际影响力分类：

### 1.1 价值分类

| 价值类别 | 数量 | 占比 | 代表任务 |
|----------|:----:|:----:|----------|
| **高价值**（发现真实 bug 或隐患） | 5 | 29% | ITM-033（MemoryPack 缺配置）、ITM-038（连接泄漏） |
| **中价值**（改善一致性但无行为变更） | 7 | 41% | ITM-025（中文注释统一）、ITM-037（editorconfig） |
| **低价值**（纯装饰/形式主义） | 3 | 18% | ITM-032（README 计数修正—后续放弃） |
| **负价值**（引入新风险） | 1 | 6% | 见下方 ITM-031 副作用 |
| **未实施/阻塞** | 1 | 6% | ITM-041（等待 .NET 11 GA） |

**结论**：17 项任务中仅 5 项（29%）真正发现并修复了问题。近半（41%）是注释/配置的一致性工作，对系统行为无影响。ITM-032 已因后续决策（接受计数漂移）而事实废弃。

### 1.2 ITM-031 的隐藏副作用（负价值案例）

ITM-031 的修改逻辑：
- **主张**：将 CA1031 从全局 NoWarn 移除，改为 27 处精确 [SuppressMessage]
- **实际执行**：只改了 5 处（带 when 过滤的 22 处不触发 CA1031，未改动）
- **表面收益**：5 个 [SuppressMessage] 比 1 个全局 NoWarn 更"精确"
- **隐藏副作用**：

```
TreatWarningsAsErrors=true 的项目中：
- 22 处 `catch(Exception ex) when(ex is not OperationCanceledException)` 无 [SuppressMessage]
- 当前 .NET 11 Preview 5 的 CA1031 不触发带 when 的 catch
- 但 Microsoft 官方文档 CA1031 条款未提及 when 子句会豁免分析器
- 若未来 .NET 版本收紧行为（对 when 过滤的 catch 也报警），
  22 处代码将突然报错 → 编译失败 → CI 阻断
```

这是本次审计发现的**评审→修复链路中最严重的副作用**：一次出于"精确性"的修改，因未回溯分析器行为规范，引入了潜在的编译失败风险。该风险当前被 `dotnet build` 通过的实证掩盖，但缺乏文档保障。

## 二、评审报告的 5 项误判全览

### 2.1 误判清单及修正方式

| ID | 误判内容 | 来源 | 修正方式 |
|----|----------|------|----------|
| **M1** | `InMemoryOutboxStore.RequeueDeadAsync` 并发竞态 | v4 评审 P2 | 源码核查→已移除 |
| **M2** | `PeriodicBackgroundProcessor` 缺 OCE 过滤 | v4 评审 P2 | 源码核查→降级 P3 |
| **M3** | `DapperProjectionCheckpointStore` 缺少实现 | v2 评审 | 自修正（"已过期"） |
| **M4** | `catch(Exception)` 异常过滤不一致（27 vs 11） | 本轮初判 | 逐行核查→全合规 |
| **M5** | CA1031 需 27 处 SuppressMessage | ITM-031 预估 | build 实测→仅 5 处 |

### 2.2 误判的共通模式

全部 5 项误判有**完全相同的根因**：**基于片段信息做判断，未获取完整上下文**。

| 误判 | 缺失的上下文 | 获取完整上下文的正确方式 |
|------|------------|----------------------|
| M1 | 未读 lock 块体的后半段 | 读完整方法体 |
| M2 | 未区分"关停取消"和"非关停取消" | 列举语义子场景后判断 |
| M3 | 基于旧版本快照 | 在当前 commit 上 grep |
| M4 | 只看了 catch(Exception) 计数的差值 | 逐行读每个 catch 的前一行 |
| M5 | 不理解 CA1031 对 when 子句的行为 | dotnet build 实测 |

**模式**：5 个误判的修复方式完全相同——"多读几行代码就修正了"。没有任何一个误判需要设计讨论或架构论证。这指向一个系统性问题：**评审报告在缺乏完整上下文的情况下就给出了定论和优先级分级，而非标注为"待验证"**。

## 三、评审报告中的过度评价

### 3.1 评分膨胀

v4 评审给出的评分分布：

| 维度 | 评分 | 同时期未发现的真实问题 |
|------|:----:|----------------------|
| 可维护性 | 9/10 | development.md 5 个无效路径（文档不可执行） |
| 健壮性 | 9/10 | PostgreSqlReadWriteRouter 连接泄漏（IAsyncDisposable 缺失） |
| 可读性 | 8/10 | 中英 summary 混杂（48 处英文残留） |
| 可扩展性 | 9/10 | MemoryPack 缺 VerifyReferenceAotCompatibility 配置 |
| 合理性 | 9/10 | CA1031 全局 NoWarn（拒绝精确抑制的全局放行） |

**模式**：得分 8-9/10 的维度下均有未发现的实质性缺陷。评审的评分标准似乎偏向"架构设计质量"（分层清晰、接口合理）而忽略了"实现完整度"（配置齐全、资源释放、文档可执行性）。

### 3.2 补充卷 II 的正确示范

`serena-review-supplement-ii.md` 是唯一聚焦**实现质量**而非架构质量的评审——它逐行审计 SQL 注入、线程安全原语、技术债标记。该卷发现了 PostgreSqlAuditor 的 SQL 注入风险（已在集成测试中验证修复），没有给"9/10"的分数——而是逐项列出"✅ 零风险""⚠️ 调用方控制"等精确判定。这是评审质量最高的单份报告。

## 四、任务清单执行层面的错误

### 4.1 ITM-031 过度预估——工作量预估偏差 5.4 倍

| 预估 | 实际 | 偏差 |
|------|------|:---:|
| "涉及 27 处源码修改" | 5 处修改 | 5.4 倍 |
| "风险：中" | 实际有 1 项隐藏副作用 | 低估而非高估 |
| "涉及 16 个文件" | 5 个文件 | 3.2 倍 |

**根因不是"数错了"而是"不理解分析器行为"**——将 `catch(Exception)的grep结果数` 等同于 `CA1031触发数`，这是对静态分析器工作原理的根本误解。

### 4.2 ITM-039 方法名事实错误——交叉审计缺失

用户提供的任务描述写 `SoftDelete`/`Restore`/`PurgeOld`，但源码实际是 `Delete`/`Restore`/`Purge`。**这个错误从任务清单编写到实施前一直未被发现**——直到 `Read` 命令打开源文件才暴露。

**根因**：任务合并（用户提供→我合并到 action-items）过程中无交叉验证步骤。如果当时用 `grep -rl "SoftDelete" src/` 快速核实，30 秒就能发现方法名不存在。

### 4.3 ITM-038 无测试护航的修复

`PostgreSqlReadWriteRouter` 的 `IAsyncDisposable` 实现修复了一个真实的连接泄漏风险——这是 17 项任务中最高价值的 2 项之一。但修复后**未新增任何测试**验证：
- Dispose 后调用 `GetReader()` 应抛 `ObjectDisposedException`
- DI 容器在应用关停时是否实际调用了 `DisposeAsync`
- `Reader` 为 `null` 时 `DisposeAsync` 不应崩溃

**这是"高价值修复但低质量交付"的典型案例**：修复本身正确，但缺乏回归测试保护，未来重构可能破坏此行为。

## 五、根本原因模型（修正版）

前几轮审计给出了 11 类根因（R1~R11），但本轮的定性分析揭示这些根因可以进一步归纳为 **3 个系统性缺陷**：

### 缺陷 1：评审的信息获取策略错误

| 表现 | 根因 |
|------|------|
| 5 项误判全部来自"片段读取" | 评审工具链（grep/serena）天然倾向碎片化输出 |
| 评分 9/10 维度下均有缺陷 | 评审聚焦"架构设计"忽略"实现完整度" |
| 27 处预估 vs 5 处实际 | 不理解分析器工具的工作原理 |

**解决方案**：评审报告必须区分"已完整审计"和"基于抽样"两类判断，禁止基于抽样给出确定结论。

### 缺陷 2：任务清单缺乏交叉验证

| 表现 | 根因 |
|------|------|
| 方法名错误跨多个环节未发现 | 无"任务描述→源码验证"步骤 |
| 无测试护航的修复被提交 | 无"代码修复→测试覆盖"的强制检查 |
| ITM-031 隐藏副作用未被识别 | 无"修改→副作用回溯"步骤 |

**解决方案**：任务清单的"验证"段落从文档约定升级为强制执行的脚本检查（`verify-action-items.sh` 已部分实现，但方法名验证需精确到 grep 确认层）。

### 缺陷 3：装饰性工作挤占了真正重要的事

| 表现 | 根因 |
|------|------|
| 17 项任务中仅 5 项高价值 | 评审→任务清单的优先级筛选不严格 |
| ITM-037（editorconfig 4 行）与 ITM-038（连接泄漏）同为 P2 | P1/P2/P3 分类仅基于"修复复杂度"而非"危害程度" |
| ITM-032 的 README 计数工作最终被废弃 | 执行时未重新评估"这项工作还值得做吗" |

**解决方案**：任务清单的优先级定义从 P1=近期修复/P2=中期增强/P3=长期演化 改为 高价值=阻止bug或数据损失/中价值=改善一致性/低价值=装饰性或可推迟。P2 不应混入纯装饰工作。

## 六、最优解决方案（全新——针对定性缺陷）

### 方案 1：评审报告增加"可信度标注"

每个评审结论必须标注为三类之一：

| 标注 | 含义 | 示例 |
|------|------|------|
| **✅ 已完整审计** | 读完整方法/类，基于完整上下文 | "Saga.cs 178 行的 catch 带 when(attempt<MaxRetries)，前有 catch(OCE)，后无遗漏" |
| **⚠ 基于抽样** | grep 或搜索命中的条目，未逐行验证 | "grep 显示 27 处 catch(Exception)，其中 11 处带 when 过滤——**其余 16 处待逐行核查**" |
| **❓ 待验证** | 基于假设或需要 build 验证 | "假设 CA1031 对所有 catch(Exception)报警，需 dotnet build 验证" |

禁止基于"⚠ 基于抽样"得出确定性结论（如"异常过滤不一致"）。

### 方案 2：任务清单增加"价值/风险双维度评分"

将 ITM 条目的优先级从一维（P1/P2/P3）改为二维：

| | 高危害 | 中危害 | 低危害 |
|------|:------:|:------:|:------:|
| **易修复** | P0 | P1 | P2 |
| **中等** | P1 | P2 | P3 |
| **难修复** | P2 | P3 | 评估 |

"危害"定义：数据损失/安全漏洞/运行时崩溃/资源泄漏 = 高危害；不一致/注释缺失/文档不准 = 中危害；纯装饰 = 低危害。

### 方案 3：修复后强制检查清单

每个代码修改任务实现后，提交前必须勾选：

- [ ] 修改是否新增了测试覆盖？（高性能 bug 修复必须有测试）
- [ ] 修改是否引入了新的编译期依赖？（如新增 `using` 或项目引用）
- [ ] 修改是否改变了现有行为的语义？（如从全局 NoWarn 改为 SuppressMessage）
- [ ] 如果改变了语义，是否回溯验证了所有受影响路径？
- [ ] 如果新增了 [SuppressMessage]，是否已确认分析器行为有文档保障？

### 方案 4：任务合并时强制交叉验证

在 AGENTS.md 或 Trellis workflow 中增加：

```markdown
## 外部任务合并规则
合并外部提供的任务到 action-items 时：
1. 任务描述中的每个方法名/类名 → grep -rl 验证存在
2. 任务描述中的每个文件路径 → ls 验证存在
3. 任务描述中的每个配置属性 → grep 验证拼写
4. 验证结果标注在任务条目末尾 "✅ 交叉验证通过"
```

## 七、终极结论

本轮定性审计的发现可以浓缩为一句：

**评审质量的主要问题不是"找错了什么问题"，而是"把太多不重要的事放到了重要位置上"——17 项任务中仅 29% 有高价值，而一项被标记为"中风险"的任务（ITM-031）引入了潜在副作用。**

具体而言：
- 5 项误判的根因完全相同（不完整读取），修复也完全相同（多读几行）
- 评审评分 9/10 的维度下均存在未发现的实质性缺陷——评分标准偏向"架构设计"忽略"实现完整度"
- 补充卷 II 是唯一聚焦实现质量的报告，质量最高——提供了评审应如何做的正面示范
- ITM-031 的隐藏副作用是评审→修复链路中最严重的失误——修改引入了潜在的编译失败风险
- ITM-038 的高价值修复因无测试护航，降低了长期维护安全性

**预防未来错误的最高杠杆点**：不是"更仔细地读代码"（这不可靠，R1 本身在审计中反复复发），而是"改变评审的信息策略"——用"可信度标注"替代"确定性判断"，用"价值/风险双维度"替代"P1/P2/P3 单维度"。这些是流程级别的改进，不依赖个体纪律。
