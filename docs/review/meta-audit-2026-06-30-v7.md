## 四、终极预防方案（增量 3 项）

现有体系已覆盖 27 项错误中的 23 项根因。需要补充的 3 项：

### 预防 1：操作路径自动化检查（预防 G1/G2 中最严重的错误）

**问题**：development.md 的 5 个无效路径直到第四轮才被发现——因为审计一直在找"数字"，而非"路径引用的正确性"。

**方案**：在 `scripts/verify-conventions.sh` 中增加文档路径引用检查：

```bash
# 新增：扫描 docs/ 中引用的项目路径是否真实存在
echo "── 文档路径引用检查 ──"
FAIL=0
for path in $(grep -rohP '(?:test|src|bench|samples)/[A-Za-z0-9_.]+/[A-Za-z0-9_.]+' docs/*.md docs/review/*.md 2>/dev/null | sort -u); do
  if [ ! -e "$path" ]; then
    echo "  ✗ docs/ 引用的路径 '$path' 不存在"
    FAIL=1
  fi
done
[ $FAIL -eq 0 ] && echo "  ✓ 全部路径有效"
```

**价值**：这种错误 4 项，是 27 项中唯一有实际危害的错误类型。自动化检查直接堵死。

### 预防 2：评审前强制运行 snapshot（整合到 review-snapshot.sh）

**问题**：review-snapshot.sh 存在但未强制——评审者可以不运行直接写报告。

**方案**：修改 `REVIEW_TEMPLATE.md`——评审报告首行必须粘贴 snapshot 输出（已经是强制要求）；同时在 conventions.md 中增加：未含 snapshot 输出的评审报告被视为草稿。

### 预防 3：AI Agent 上下文注入（从源头减少 G2/G3 错误）

**问题**：G2（未验证采信）和 G3（认知盲区）的核心原因是 AI Agent 在评审时"凭记忆"而非"凭实测"。

**方案**：在 AGENTS.md 评审相关段落增加强制规则：

```markdown
## 评审强制规则（元审计预防）

进行任何代码评审时：
1. 先运行 `bash scripts/review-snapshot.sh`，将输出粘贴到报告首行
2. 任何数字声明必须标注来源（实测/估算/引用）
3. 任何"分析器规则会触发"的声明必须先 `dotnet build` 验证
4. 任何 grep 计数差异必须逐项核查上下文（禁止仅凭数字下结论）
5. 任务清单中引用的方法名/类名必须 `grep -rl` 验证存在
```

## 五、终极结论

### 5.1 数字总结

| 指标 | 值 |
|------|:--:|
| 四轮审计发现的错误总数 | 27 |
| 有实际危害的操作错误 | 4（已全部修复） |
| 无危害的装饰性漂移 | 23（已接受，不再追踪） |
| 评审流程自身的失误 | 9（已自纠，预防机制已部署） |
| 根因分类 | G1 不完整读取 / G2 未验证采信 / G3 认知盲区 |
| 现有预防机制 | 6 项（review-snapshot/verify-action-items/架构测试/评审模板/评审纪律/verify-conventions） |
| 增量预防方案 | 3 项（路径自动化检查/强制snapshot/AI Agent上下文注入） |

### 5.2 核心洞察

**四轮审计的最关键发现不是"有 27 项错误"，而是"27 项中仅 4 项有实际危害"——另外 23 项是装饰性数字漂移，不影响任何读者操作或决策。**

这意味着：
1. **前几轮审计投入过度的精力在装饰性漂移上**——这是一个被 verify-doc-numbers.sh 的存在放大了的"假问题"
2. **真正重要的错误（路径引用）直到第四轮才被发现**——因为审计工具（grep 计数）天然倾向找数字而非路径
3. **最优预防不是"更多校验"，而是"只校验有害的东西"**——删除 verify-doc-numbers.sh 是正确的，增加路径引用自动化检查才是关键
