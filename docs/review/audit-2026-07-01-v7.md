### 2.2 P2 — 计划修复

#### ITM-051 · 示例代码与框架推荐路径不一致 [可信度 ✅]

- **维度**：可复用性 / 消费者视角
- **危害**：中——消费者按示例学习，但示例未使用框架推荐的最佳实践
- **证据**：

`samples/PalDDD.MinimalApi/Program.cs`:
```csharp
// 示例用了低层 API：
builder.Services.AddSingleton<ICommandHandler<CreateOrderCmd, OrderId>, CreateOrderH>();
d.Register<CreateOrderCmd, OrderId, CreateOrderH>();

// 但文档和 AGENTS.md 推荐的是：
builder.Services.AddPalCommandHandler<CreateOrderCmd, OrderId, CreateOrderH>();
```

`samples/PalDDD.ECommerce/Program.cs`:
- 未使用 `[BoundedContext]`（PDDD001 会报错）
- 未使用 `[GenerateMessage]`（PALMSG001 会报错）
- 未展示 Outbox/Inbox/Saga 集成（框架核心能力）

- **论证**：示例是消费者的第一接触点。如果示例不遵循框架推荐的最佳实践，消费者会学到错误的模式。MinimalApi 示例用 `AddSingleton+Register` 而非 `AddPalCommandHandler`，ECommerce 示例缺失战略 DDD 标记——这意味着**消费者照抄示例会遇到编译错误或漏掉核心能力**。
- **修复**：重写两个示例，使用框架推荐的完整 API 路径
- **权衡**：示例重写需要时间，但这是消费者体验的核心

### 2.3 P3 — 评估

#### ITM-052 · AOT 路径 vs 功能完整性的消费者引导缺失 [可信度 ✅]

- **维度**：兼容性 / 消费者视角
- **证据**：AOT 路径仅支持 Dapper（7 项目），EFCore/Kafka/RabbitMQ 全部 `IsAotCompatible=false`（7 项目）。消费者选择 AOT 路径时失去 EF Core 的便利性；选择 EFCore 路径时失去 AOT 发布能力。这个**核心取舍**在 README 和 architecture.md 中有提及但未突出。
- **建议**：在 README "快速开始"段增加"AOT 路径 vs 功能路径"决策树

#### ITM-053 · 包依赖图复杂度——PalDDD.Prompts 引用 9 个项目 [可信度 ✅]

- **维度**：可复用性 / 简洁性
- **证据**：`PalDDD.Prompts` metapackage 引用 9 个项目，是依赖最重的包。消费者引用 `PalDDD.Prompts` 会拉入全部依赖（包括不需要的 Kafka/RabbitMQ）。
- **评估**：设计合理（metapackage 的目的就是全量引用），但应在 XML doc 中标注"仅推荐快速原型使用，生产环境应按需引用"

#### ITM-054 · 31 个项目的平均文件数偏低（5.4 文件/项目）[可信度 ✅]

- **维度**：简洁性 / 可维护性
- **证据**：168 源文件 / 31 项目 = 5.4 文件/项目。ADR-012 已评估"维持现状"。
- **评估**：已在 ADR-012 论证合理。无需修改。

## 第三部分：10 维度设计取舍论证

### 可维护性 9/10

**优点**：分层清晰（Domain→App-Abstractions→App-Core→Infra→Hosting），约定文档化（conventions.md 13 章），ADR 完备（14 份）。

**取舍**：31 个项目对新手导航有挑战，但 slnx 分层解决了这个问题。命名守护（3 个架构测试）确保命名一致性。

**改进空间**：文档与示例的不一致（ITM-051）降低可维护性评分。

### 健壮性 10/10

**优点**：27 处 catch(Exception) 全部 OCE 合规，15 处 [SuppressMessage] 精确抑制，IAsyncDisposable 完整实现。

**取舍**：ITM-031 的隐藏副作用（22 处 when 过滤的 catch 无 SuppressMessage）——概率低但存在。

**改进空间**：无。

### 可读性 9/10

**优点**：Emoji 语义化注释，文件头 `═══`/`───` 分隔，XML doc 中文 summary。

**取舍**：中英混杂在部分长尾属性上残留（ITM-025 已修复大部分）。

**改进空间**：无重大。

### 可扩展性 9/10

**优点**：双 ORM（Dapper 7 项目 + EFCore 5 项目）+ 三方言（PG/MySQL/SQLite）+ 双 Broker（Kafka/RabbitMQ）。统一接口（IPalOutboxStore 9 方法 / IEventLog 1 方法）。

**取舍**：双 ORM 导致接口需同时满足 Dapper 和 EFCore 的约束——`IPalOutboxStore` 的同步方法（MarkProcessed/MarkDead）对 EFCore 不自然（EFCore 习惯 async SaveChanges）。这是 Dapper AOT 路径的必要妥协。

**改进空间**：IPalOutboxStore 的同步方法设计可文档化说明"Dapper 同步路径优先，EFCore 通过 SaveChangesAsync 桥接"。

### 灵活性 8/10

**优点**：15 条 PDDD 编译期规则 + 9 个 Options 类——编译期强制 + 运行时配置的平衡。

**取舍**：编译期治理严格（不提供 IRepository<T>、不提供 EventBus、不提供 IUpcaster）——这限制了"框架帮你决定"的灵活性，但换来了"架构不可腐化"的确定性。

**改进空间**：无——YAGNI 原则的正确应用。

### 简洁性 9/10

**优点**：24 条"不做"约束，每条有理由。删除了 IRepository<T>/EventBus/IUpcaster/ContextCarrier。

**取舍**：5.4 文件/项目偏低，但 ADR-012 论证合理（按需引用价值 > 项目数简洁性）。

**改进空间**：无。

### 合理性 9/10

**优点**：14 份 ADR 覆盖全部关键决策。19 处关键取舍描述。性能契约明确（FrozenDictionary/ref struct/PipelineStateMachine）。

**取舍**：.NET 11 Preview 锁定（ADR-005）——放弃 .NET 8/9/10 兼容换取 AOT 静态特性。

**改进空间**：AOT vs 功能完整性的消费者引导（ITM-052）。

### 兼容性 9/10

**优点**：AOT 配置完整（12 false 项目三属性齐全）。Public API 快照治理。

**取舍**：AOT 路径仅 Dapper，EFCore 路径非 AOT——消费者需理解这个核心取舍。

**改进空间**：README 缺少"AOT 路径 vs 功能路径"决策树（ITM-052）。

### 可复用性 8/10

**优点**：按需引用设计，每个能力独立包。NuGet metapackage 整合。

**取舍**：示例未展示完整 API 路径（ITM-051）——消费者无法从示例学习最佳实践。

**改进空间**：示例重写 + PalDDD.Prompts 文档标注使用场景（ITM-051/053）。

### 可测试性 10/10

**优点**：测试金字塔健康（架构测试 74 + 单元 378 + 集成 129 + Broker 6）。Stryker 突变测试。动态扫描架构测试。

**取舍**：无。

**改进空间**：无。

### 综合评分：8.6/10
