# 任务清单模板

> 基于专业审计报告（`docs/review/professional-audit-2026-06-30.md`）的优先级改革。
> 替代旧 P1/P2/P3 单维度分类，采用 **危害 × 复杂度** 双维度判定。

---

## 优先级矩阵

| | 高危害 | 中危害 | 低危害 |
|------|:------:|:------:|:------:|
| **易修复**（< 1h） | P0 紧急 | P1 近期 | P2 |
| **中等**（1-4h） | P1 近期 | P2 | P3 |
| **难修复**（> 4h 或需设计讨论） | P2 | P3 | 评估（产出 ADR 结论即可） |

**危害定义**：
- **高危害**：数据损失 / 安全漏洞（SQL 注入等）/ 资源泄漏 / 运行时崩溃 / 编译失败
- **中危害**：不一致 / 潜在并发风险 / 架构退化 / 文档命令不可执行
- **低危害**：注释缺失 / IDE 噪音 / 装饰性不一致 / 纯计数漂移

---

## 任务条目格式

```markdown
### [ ] ITM-XXX · 标题 · 可信度 ✅/⚠/❓
- **维度**：健壮性 / 可维护性 / 可读性 / ...
- **优先级**：[P0/P1/P2/P3/评估] · 危害: [高/中/低] · 复杂度: [易/中/难]
- **问题**：[引用代码时必须读完整方法体，禁止基于片段下结论]
- **建议**：
- **风险**：
- **验证**：
  - 涉及分析器规则 → 须附 `dotnet build` 命令
  - 涉及数字声明 → 须附实测命令
  - 涉及方法名/类名/路径 → 须标注 "✅ 已 grep 验证存在"
- **涉及文件**：
```

---

## 外部任务合并检查清单

合并外部（用户/其他 Agent）提供的任务到本清单时，**必须逐项完成以下验证**：

- [ ] 任务描述中的每个方法名 → `grep -rl "方法名" src/` 验证存在
- [ ] 任务描述中的每个类名 → `grep -rl "class 类名" src/` 验证存在
- [ ] 任务描述中的每个文件路径 → `ls` 验证存在
- [ ] 任务描述中的每个配置属性 → `grep` 验证拼写
- [ ] 验证结果标注在任务条目"验证"段 "✅ 交叉验证通过"
- [ ] 若方法名/类名不存在 → 修正后标注 "⚠ 原始方法名为 X，修正为 Y"

**违规案例警示**：
> ITM-039 中 `SoftDelete`/`PurgeOld` 方法名实际为 `Delete`/`Purge——因合并时未 grep 验证，错误从任务清单持续到实施前才暴露。此后的外部任务合并**必须**执行上述清单。

---

## 修复后检查清单

每个代码修改任务实施完成后，提交前：

- [ ] 修改是否引入了新的编译期依赖？（如新增 `using` 或项目引用）→ 检查 .csproj diff
- [ ] 修改是否改变了现有行为的语义？（如从全局 NoWarn 改为 SuppressMessage）→ 检查受影响路径
- [ ] 高危害修复是否有测试覆盖？（资源泄漏 / 安全漏洞 / 并发风险 必须有测试）
- [ ] 是否回溯验证了分析器行为有文档保障？（如新增 SuppressMessage 前确认官方文档行为）
